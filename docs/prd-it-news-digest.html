제품 요구사항 문서 (PRD)

제품명: IT 뉴스 트렌드 · 관심사 맞춤 요약 에이전트
문서 버전: 1.2 · 기준일: 2026-04-10
Seed 기반 인터뷰 반영 MVP 410ai

인터뷰 · 핵심 요청(oOo interview): LangGraph로 여러 도구를 쓰고, 중간 결과에 따른 분기·반복이 있는 AI 에이전트 워크플로를 Streamlit으로 구현한다. 그래프 구조와 에이전트 패턴을 제품 단계에서 명확히 기획한다.

1. 개요

공개 RSS·IT 뉴스 피드를 수집하고, 사용자의 구조화된 관심사(역할·기술 스택·관심 토픽)와의 관련도를 계산한다. 관련도가 낮으면 워크플로 분기로 피드 범위를 확장한 뒤 동일 수집 노드로 루프하여 재수집한다(MVP는 단일 확장 라운드). 정렬·점수가 확정되면 한국어 제목·요약 정규화(LLM 일괄 처리)를 거친 다음, LangGraph ReAct 서브그래프에서 도구를 호출하며 근거를 확인하고 한국어 트렌드·맞춤 요약을 제공한다. 클라이언트는 Streamlit 단일 엔트리 앱으로 제공하며, 그래프 스트리밍으로 진행 상태를 노출한다.

2. LangGraph 그래프 구조 · 에이전트 패턴 기획

본 절은 인터뷰에서 요구한 「다중 도구 · 중간 결과 분기/반복 · Streamlit 연동」을 만족시키기 위한 토폴로지와 패턴 선택 근거를 정의한다.

2.1 설계 원칙

  • 상위 그래프: 데이터 수집·점수화·정책 분기·(선택) 전처리는 결정적·반복 가능한 노드로 둔다.
  • 조건부 분기: 중간 상태(예: max_match_score, scope_expanded)만으로 다음 노드를 고른다.
  • 반복(루프): 저매칭이면 피드 집합을 넓힌 뒤 수집 노드로 되돌아가 같은 파이프라인을 한 번 더 탄다.
  • ReAct 서브그래프: 자유 기획 요약·역할 수행은 create_react_agent로 캡슐화하고, 상태 스키마로 상위와 동일한 컨텍스트를 공유한다.
  • 도구: 외부 검색이 아니라 이미 수집된 상태(기사 목록·프로필)를 읽는 도구 중심으로 환각을 줄인다(상태 주입).

2.2 MVP 노드·엣지 (구현 기준: streamlit_app/workflow.py)

START
  → ingest_rss          (다중 피드 HTTP 수집·중복 제거)
  → score_articles      (프로필 대비 관련도·정렬)
  → [조건] route_after_score
        • (최고 관련도 < 임계값) ∧ (아직 피드 미확장)
              → expand_scope → **루프** → ingest_rss
        • 그 외
              → localize_articles (상위 N건 한국어 제목·요약)
              → prepare_react     (사용자 메시지 시드)
              → react_agent       (ReAct: 도구 루프)
              → END

분기 조건은 코드상 MATCH_THRESHOLD와 scope_expanded 플래그로 단순화한다. 확장 후에도 임계값 미만이면 추가 루프 없이 다음 단계로 진행(MVP 정책).

2.3 상태(Shared State)와 도구의 역할

구분역할
공유 상태 articles, article_scores, user_profile, feed_urls, branch_events, tool_trace, ReAct용 messages
도구 (ReAct) 예: 관련도 순 헤드라인 조회, 사용자 관심사 프로필 문자열 조회 — 모두 InjectedState로 그래프 상태를 읽음
Streamlit graph.stream(..., stream_mode="values")로 노드 진행에 맞춰 tool_trace·분기 안내·메시지를 갱신

2.4 패턴 요약

패턴적용목적
조건부 엣지score_articles 이후중간 결과(관련도·확장 여부)에 따른 분기
루프백expand_scope → ingest_rss범위 확장 후 재수집·재점수화
ReAct 서브그래프react_agent 노드LLM이 도구를 여러 번 호출하며 최종 한국어 큐레이션 생성
전처리 LLM 노드localize_articles다국어 RSS를 사용자-facing 한국어로 통일(ReAct 입력 품질 향상)

2.5 후속 확장 아이디어 (비-MVP)

  • 임계값 미만 시 다회 확장 또는 키워드·기간 완화 노드 추가.
  • 독립 서브그래프 병렬 실행(예: 출처별 요약 병합).
  • 휴먼 인더 루프: 확장 전 사용자 확인 노드.

3. 비전 · 목표

3.1 제품 비전

IT 뉴스 홍수 속에서 사용자의 직무·기술 관점에 맞는 신호만 빠르게 요약·연결해 준다.

3.2 핵심 목표

  • RSS 기반으로 API 키 의존을 최소화한 뉴스 수집.
  • 관심사 대비 저매칭 시 자동으로 검색 범위(피드) 확장을 정책에 따라 한 번까지 수행한다.
  • ReAct로 다중 도구를 사용하고, UI에서 도구 타임라인·분기 이유를 노출해 신뢰도와 설명 가능성을 확보한다.
  • LangGraph 상위 그래프에서 분기·루프를 두어 중간 결과에 따른 동작을 제품 요구사항 수준에서 설명 가능하게 한다.
  • 모든 사용자 대면 콘텐츠는 한국어 위주로 통일한다.

4. 사용자 · 시나리오

4.1 주요 사용자

IT 업계 종사자 또는 학습자로, 특정 스택·토픽에 맞춰 뉴스를 걸러 요약받고 싶은 사람.

4.2 대표 사용자 스토리

ID스토리비고
US-1 역할·스택·토픽을 입력하고 한 번의 실행으로 최신 IT 뉴스 기반 요약을 받고 싶다. 구조화 입력
US-2 내 관심사와 뉴스의 연결이 약할 때, 시스템이 범위를 넓혀 다시 시도하는 것을 이해하고 싶다. 분기·루프 UI

US-5 읽은 흐름과 관심사에 맞춰 인프런에서 이어서 배울 강의·검색을 추천받고 싶다. get_inflearn_learning_suggestions

5. 범위

5.1 MVP 포함

  • streamlit run 단일 엔트리.
  • 기본·확장 RSS 피드 집합 구성(구체 URL은 구현에서 관리, 변경 가능).
  • LangGraph: 수집 → 점수화 → 조건부 확장(루프) → 한국어 메타 정규화 → ReAct 요약 서브그래프.
  • 도구: 수집된 기사 목록·사용자 프로필 조회 등(상태 주입 패턴).
  • UI: 기사 제목(외부 링크)·짧은 요약, AI 요약, 분기 안내, 타임라인(접기).

5.2 비목표 (MVP 제외)

  • 사용자 계정·로그인·멀티 테넌트.
  • DB 영속화, 즐겨찾기 동기화.
  • 이메일·푸시 리포트.
  • 특정 유료 뉴스 API를 MVP 필수 경로로 두지 않음(RSS 기본, NewsAPI는 선택).

7. 비기능 요구사항

  • 모델: 채팅은 gpt-5-mini 고정. 샘플링 파라미터 temperature / top_p 미사용(해당 계열 제약).
  • 임베딩(선택): 필요 시 text-embedding-3-small — MVP 점수는 규칙 기반 가능.
  • 설정: OPENAI_API_KEY는 환경 변수 또는 .env(프로젝트 루트·노트북·앱 디렉터리 등 합의된 경로).
  • 신뢰성: 개별 피드 실패 시 가능한 한 나머지 피드로 계속, 오류는 로그에 반영.
  • 보안: API 키는 저장소에 커밋하지 않음.

9. 수용 기준 (Acceptance Criteria)

  • 단일 Streamlit 엔트리로 앱이 기동된다.
  • 실행 시 RSS에서 기사가 수집되고, 관련도에 따라 확장 분기와 수집 루프가 0 또는 1회 발생할 수 있다.
  • 최종 화면에 한국어 요약과 기사 링크·한국어 위주 짧은 요약이 보인다.
  • 저매칭 확장이 일어나면 사용자에게 그 이유가 표시된다.
  • ReAct 구간의 도구 사용이 타임라인으로 확인 가능하다.
  • PRD 2절에 기술한 그래프 토폴로지와 구현이 정합적이다(노드·분기·루프).

11. 잔여 결정 사항 · 리스크

열린 결정: RSS 화이트리스트·갱신 주기, 매칭 임계값·점수 정의(규칙 vs 임베딩), 확장 시 정책(추가 피드만 / 기간·키워드 완화 등)의 세부 규칙.
  • 리스크: RSS 제공처 변경·차단 → 피드 목록 유지보수 필요.
  • 리스크: LLM 환각 → 도구로 근거 확인 및 UI에 출처 노출.
